## UX Flows

### Primary Journey: Public visitor reads a debate

**Desktop (≥1024 px)**

| Step | Description |
|---|---|
| Entry | Navigate to `/` in any modern browser |
| 1 | Page loads. Debate topic heading (`h1`) is immediately visible, centered at top |
| 2 | Below topic: a sticky legend bar reading "Tark · for · Vitark · against" with colored dots identifies the two sides |
| 3 | Below legend: a timeline with a vertical center spine. Tark (blue) cards are positioned left of the spine, Vitark (amber) cards right of the spine, arranged top-to-bottom in posting order. Cards carry argument text only — no per-card side labels |
| 4 | Visitor reads the debate thread top-to-bottom; page scrolls naturally if content exceeds viewport |
| Exit | Browser close, back navigation, or idle — no in-page actions |

**Tablet (~768 px)**

| Step | Description |
|---|---|
| Entry | Same as desktop |
| 1 | Topic heading visible, centered |
| 2 | Timeline layout preserved with narrower margins and card widths |
| 3 | Natural page scroll for overflow |
| Exit | Same as desktop |

**Mobile (≤480 px)**

| Step | Description |
|---|---|
| Entry | Same as desktop |
| 1 | Topic heading visible, full-width |
| 2 | A sticky legend bar ("Tark · for · Vitark · against" with colored dots) sits below the topic heading |
| 3 | Arguments display as chat-style cards in posting order. Tark (blue) cards are left-aligned, Vitark (amber) cards are right-aligned, creating a conversational back-and-forth feel. Cards carry argument text only — no per-card side labels |
| 4 | Visitor scrolls vertically to read the debate thread |
| Exit | Same as desktop |

### System Journey: Static render

| Step | Description |
|---|---|
| Entry | Browser requests root path |
| 1 | JS framework renders component tree with embedded static data |
| 2 | Theme resolved from `data-theme` attribute or `prefers-color-scheme` |
| 3 | All content renders in correct theme variant |
| Exit | Render complete; no runtime state |

## Interaction Notes

### Content and hierarchy

- **Visual hierarchy**: Topic (`h1`, largest) > Legend bar (sticky, identifies sides) > Argument text (body)
- **Legend bar**: A sticky bar below the header displays "● Tark · for · ● Vitark · against" with colored dots matching card fill colors. Per-card "Tark"/"Vitark" text labels are removed — side identity is conveyed through color coding + position + the legend
- **Representative count**: ~8 arguments total (mix of Tark and Vitark). Layout must accommodate 4–12 gracefully
- **Argument length**: 1–3 sentences each. Layout must handle variable lengths without alignment issues

### Layout

- **Desktop/tablet**: Timeline with vertical center spine. Topic spans full width above. Tark cards positioned left of spine, Vitark cards right of spine. Cards are arranged top-to-bottom in posting order
- **Mobile**: Single column, chat-bubble style. Topic at top, legend bar below, then cards stacked in posting order. Tark cards are left-aligned, Vitark cards are right-aligned (~72% width), creating a WhatsApp-like conversational stagger. Each card is color-coded (blue for Tark, amber for Vitark)
- **Card shape**: Chat-bubble style — asymmetric corner radii (sharp 4px corner on the speaker side, 12px rounded on the other three corners) with a small triangular tail (8×10px) protruding outward from the sharp-corner edge. Tark cards have sharp top-left + left-pointing tail; Vitark cards have sharp top-right + right-pointing tail
- **Overflow strategy (OQ-2 resolved)**: Natural page scroll. No pagination. No independent scrollable containers. No scroll traps
- Arguments render as distinct visual cards, each tagged with its side (Tark/Vitark)
- Cards are independently sized — no cross-alignment requirement between left and right positions

### Accessibility (FR-9)

- **Semantic HTML**: `<main>`, `<article>` or `<section>` for debate, `<h1>` for topic, `<h2>` for section labels, `<ul>`/`<li>` or equivalent for argument lists
- **Reading/DOM order**: Topic → Arguments in posting order (each card carries its own side label)
- **Keyboard**: Tab navigates through landmark regions. No focusable interactive controls (read-only). Skip-link to main content optional at Gate 4
- **Contrast**: All text meets WCAG 2.1 AA — 4.5:1 for normal text, 3:1 for large text (≥18px or ≥14px bold) in both themes
- **ARIA**: `role="main"` on content area, section labels associated with their argument groups via heading hierarchy. No `aria-live` needed (static content)
- **Legend bar a11y**: Each argument card retains an `aria-label` (e.g., "Tark argument" / "Vitark argument") so screen readers announce side identity even without visible per-card text labels
- **Screen reader**: Announcement order matches DOM order. Sections are navigable by heading level

### Theme

- Light and Dark variants for every screen. Token-based — no hardcoded colors in implementation
- Theme switch is CSS-only (instant, no JS transition logic needed)
- Accent colors for Tark (blue family) and Vitark (amber family) adjust for readability in both themes (Known Rule #40: no value-laden color pairs)

### Content dependencies

- Placeholder debate content acceptable for design layouts
- Suggested placeholder topic: "Should artificial intelligence be regulated by international law?"
- Final debate content determined by Gate 5 implementation
