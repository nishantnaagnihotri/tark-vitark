## UX Readiness

**Ready** — Challenge Phase complete, all gaps resolved, OQ-2 resolved.

## UX Flows

### Primary Journey: Public visitor reads a debate

**Desktop (≥1024 px)**

| Step | Description |
|---|---|
| Entry | Navigate to `/` in any modern browser |
| 1 | Page loads. Debate topic heading (`h1`) is immediately visible, centered at top |
| 2 | Below topic: a sticky legend bar reading "Tark · for · Vitark · against" with colored dots identifies the two sides |
| 3 | Below legend: a timeline with a vertical center spine. Tark (blue) cards are positioned left of the spine, Vitark (amber) cards right of the spine, arranged top-to-bottom in posting order. Cards carry argument text only — no per-card side labels |
| 4 | Visitor reads the debate thread top-to-bottom; page scrolls naturally if content exceeds viewport |
| Exit | Browser close, back navigation, or idle — no in-page actions |

**Tablet (~768 px)**

| Step | Description |
|---|---|
| Entry | Same as desktop |
| 1 | Topic heading visible, centered |
| 2 | Timeline layout preserved with narrower margins and card widths |
| 3 | Natural page scroll for overflow |
| Exit | Same as desktop |

**Mobile (≤480 px)**

| Step | Description |
|---|---|
| Entry | Same as desktop |
| 1 | Topic heading visible, full-width |
| 2 | A sticky legend bar ("Tark · for · Vitark · against" with colored dots) sits below the topic heading |
| 3 | Arguments display as chat-style cards in posting order. Tark (blue) cards are left-aligned, Vitark (amber) cards are right-aligned, creating a conversational back-and-forth feel. Cards carry argument text only — no per-card side labels |
| 4 | Visitor scrolls vertically to read the debate thread |
| Exit | Same as desktop |

## Interaction Notes

### Content and hierarchy

- **Visual hierarchy**: Topic (`h1`, largest) > Legend bar (sticky, identifies sides) > Argument text (body)
- **Legend bar**: A sticky bar below the header displays "● Tark · for · ● Vitark · against" with colored dots matching card fill colors. Per-card "Tark"/"Vitark" text labels are removed — side identity is conveyed through color coding + position + the legend
- **Representative count**: ~8 arguments total (mix of Tark and Vitark). Layout must accommodate 4–12 gracefully
- **Argument length**: 1–3 sentences each. Layout must handle variable lengths without alignment issues

### Layout

- **Desktop/tablet**: Timeline with vertical center spine. Topic spans full width above. Tark cards positioned left of spine, Vitark cards right of spine. Cards are arranged top-to-bottom in posting order
- **Mobile**: Single column, chat-bubble style. Topic at top, legend bar below, then cards stacked in posting order. Tark cards are left-aligned, Vitark cards are right-aligned (~72% width), creating a WhatsApp-like conversational stagger. Each card is color-coded (blue for Tark, amber for Vitark)
- **Mobile card text clamp**: Argument cards on mobile collapse to 4 lines (96px, 4×24px LH) by default. A "Read more" tap target (Inter Medium 12px, `--color-brand-primary` — consistent CTA affordance, M3 `primary` role, WCAG AA ≥4.5:1 on all card surfaces in both modes) appears below the card text. No "Show less" toggle — tapping "Read more" expands to full text. Desktop/tablet: no clamp, full text always visible
- **Card structural composition (mobile)**: Bubble tail and "Read more" are children of their respective card frame (absolute tail + auto-layout RM flex child), not siblings in the parent content frame. Both Light and Dark frames follow the same structural pattern.
- **Card shape**: Chat-bubble style — asymmetric corner radii (sharp 4px corner on the speaker side, 12px rounded on the other three corners) with a small triangular tail (8×10px) protruding outward from the sharp-corner edge. Tark cards have sharp top-left + left-pointing tail; Vitark cards have sharp top-right + right-pointing tail
- **Overflow strategy (OQ-2 resolved)**: Natural page scroll. No pagination. No independent scrollable containers. No scroll traps
- Arguments render as distinct visual cards, each tagged with its side (Tark/Vitark)
- Cards are independently sized — no cross-alignment requirement between left and right positions

### Accessibility (FR-9)

- **Semantic HTML**: `<main>`, `<article>` or `<section>` for debate, `<h1>` for topic, `<h2>` for section labels, `<ul>`/`<li>` or equivalent for argument lists
- **Reading/DOM order**: Topic → Arguments in posting order (each card carries its own side label)
- **Keyboard**: Tab navigates through landmark regions. No focusable interactive controls (read-only). Skip-link to main content optional at Gate 4
- **Contrast**: All text meets WCAG 2.1 AA — 4.5:1 for normal text, 3:1 for large text (≥18px or ≥14px bold) in both themes
- **ARIA**: `role="main"` on content area, section labels associated with their argument groups via heading hierarchy. No `aria-live` needed (static content)
- **Legend bar a11y**: Each argument card retains an `aria-label` (e.g., "Tark argument" / "Vitark argument") so screen readers announce side identity even without visible per-card text labels
- **Screen reader**: Announcement order matches DOM order. Sections are navigable by heading level

### Theme

- Light and Dark variants for every screen. Token-based — no hardcoded colors in implementation
- Theme switch is CSS-only (instant, no JS transition logic needed)
- Accent colors for Tark (blue family) and Vitark (amber family) adjust for readability in both themes (Known Rule #40: no value-laden color pairs)

### Content dependencies

- Placeholder debate content acceptable for design layouts
- Suggested placeholder topic: "Should artificial intelligence be regulated by international law?"
- Final debate content determined by Gate 5 implementation

## Challenge Phase Findings

| ID | Challenge | Classification | Finding | Resolution |
|---|---|---|---|---|
| CH-1 | **Over-engineering: Surface primitive** | Accept With Risk | LegendBar uses a themed container. Could define a Surface primitive. However, the container is only used in LegendBar in this slice, with feature-specific layout (sticky positioning, inline dot+text layout). No other feature component needs a generic Surface. | **Not defined.** LegendBar container is feature-specific CSS with token color bindings. Revisit if a future slice reuses surface containers. |
| CH-2 | **Over-engineering: Dot/Indicator primitive** | Accept With Risk | Colored dots appear in LegendBar (side indicators) and Timeline (spine markers). Could define a DotIndicator primitive. However, these are 8px CSS circles — trivial styling, no variants, no states. Promoting to a DS primitive adds abstraction with no reuse benefit in this slice. | **Not defined.** Dots are feature-specific CSS (`width: 8px; height: 8px; border-radius: 50%`) with token color bindings. |
| CH-3 | **M3 mapping: Card tail** | Accept With Risk | ArgumentCard has a triangular tail (8×10px). M3 Filled Card has no tail concept. Could force-fit as a Card primitive shape extension. | CH-5 | **M3 mapping: Divider for timeline spine** | Accept | M3 Divider is 1dp, uses `outline-variant` color. Timeline spine is a vertical line, potentially 2px for visual presence. | CH-6 | **Slice scope: Typography roles** | Accept | M3 type scale has 15 baseline + 15 emphasized roles. This slice needs only 3 roles: headline-large (Topic), body-large (ArgumentCard text), label-medium (LegendBar). | CH-7 | **Scope: Interactive states** | Accept | M3 Filled Card defines hovered, focused, pressed, dragged, disabled states. This slice is read-only — no interactive affordances per FR-6/AC-6. Implementing interactive states would add unused complexity. | **Enabled state only.** Interactive states deferred until a future slice adds card interactivity. | CH-8 | **Missing token: on-surface** | Accept | Architecture §2.6 defines per-component on-surface colors but omits a general `--color-on-surface` token for page-level text (e.g., Topic heading). ## Feature Component → DS Primitive Decomposition

### ArgumentCard

| Layer | Source | Description |
|---|---|---|
| Container | **Card (Filled)** primitive | Provides shaped rectangle with per-corner radii, fill color role, and internal padding. Variant (`tark` or `vitark`) selects color scheme and corner assignment. |
| Text content | **Typography** primitive (`body-large` role) | Renders 1–3 sentence argument text. Color bound to Card variant's on-surface token. |
| Triangular tail | Feature-specific CSS | `::before` pseudo-element with `clip-path: polygon()`. 8×10px. Positioned at the sharp-corner edge, pointing outward toward spine (left for tark, right for vitark). Fill color matches Card surface. Not part of Card primitive. |
| Aria label | Feature-specific HTML | `aria-label="Tark argument"` or `aria-label="Vitark argument"` for screen reader side identification. |

**Composition rule:** `ArgumentCard` renders a `Card` with a `Typography` child. The tail pseudo-element and aria-label are ArgumentCard's own additions.

### Topic

| Layer | Source | Description |
|---|---|---|
| Heading text | **Typography** primitive (`headline-large` role) | `<h1>` element, largest text on page. Color bound to `--color-on-surface`. |
| Layout | Feature-specific CSS | Centered alignment, full-width, margin-bottom spacing to legend bar. |

**Composition rule:** `Topic` is a thin wrapper — a centered `<h1>` using the `headline-large` Typography role. No other DS primitives involved.

### LegendBar

| Layer | Source | Description |
|---|---|---|
| Label text | **Typography** primitive (`label-medium` role) | "Tark", "for", "Vitark", "against" text segments. Weight 500 per M3 label-medium spec. |
| Colored dots | Feature-specific CSS | Inline `<span>` elements styled as 8px circles. Tark dot uses `--color-tark-surface`; Vitark dot uses `--color-vitark-surface`. |
| Separator | Feature-specific CSS | Visual separator between Tark and Vitark groups. CSS border or character. Uses `--color-legend-separator`. |
| Container | Feature-specific CSS | `<nav>` element with `position: sticky`, background `--color-legend-surface`, text color `--color-legend-on-surface`. Not a DS Surface primitive — layout and behavior are unique to LegendBar. |

**Composition rule:** `LegendBar` uses only the `label-medium` Typography role from DS primitives. Container, dots, separator, and sticky behavior are all feature-specific.

### Timeline

| Layer | Source | Description |
|---|---|---|
| Center spine line | **Divider** primitive (`vertical`) | Continuous vertical line running the height of the argument list. Color bound to `--color-spine-line`. Placed in center grid column (desktop/tablet) or hidden (mobile). |
| Dot markers | Feature-specific CSS | `::before` pseudo-elements on list items, 8px circles at each card's vertical position. Color `--color-spine-dot`. Decorative only. |
| Argument cards | **Card (Filled)** primitives via `ArgumentCard` | Each `<li>` contains an `ArgumentCard` composed from Card + Typography primitives. |
| Layout grid | Feature-specific CSS | Desktop/tablet: CSS Grid with 3 columns (left cards \| spine \| right cards). Mobile: single-column flex with left/right alignment stagger at ~72% width. |
| Ordered list | Feature-specific HTML | `<ol>` wrapping `<li>` elements in posting order. Semantic ordering matches DOM order. |

**Composition rule:** `Timeline` composes the `Divider` primitive (spine) and multiple `ArgumentCard` instances (which themselves compose `Card` + `Typography`). Layout grid, dot markers, and responsive breakpoint behavior are feature-specific.

### 7. Implementation Notes for Dev Agent (Gate 5)

- **Framework:** React 19 + Vite + TypeScript. React 19 passes `ref` as a standard prop to function components, making `forwardRef` unnecessary.
- **DS primitives are pure React components** implementing M3 spec independently. `@material/web` is a read-only reference, not a runtime dependency (Known Rule #51).
- **Token-first:** All visual properties come from CSS custom properties. Zero hardcoded colors/sizes in component code.
- **Primitive component API (draft):**
  - `Typography`: renders a text element with `role` prop selecting type scale (applies CSS class with token bindings). E.g., `<Typography role="headline-large" as="h1">`.
  - `Card`: renders a container with `side` prop (`"tark" | "vitark"`) selecting variant styles. E.g., `<Card side="tark">`.
  - `Divider`: renders a vertical line element. E.g., `<Divider orientation="vertical" />`.
- **Feature components compose primitives** and add their own feature-specific CSS (tail, dots, layout, sticky behavior).
