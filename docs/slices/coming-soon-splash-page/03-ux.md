## 2) UX Flows

Primary Journey A: Public visitor lands on splash page (desktop)
- Entry: Direct URL open in desktop browser at 1280px representative viewport.
- Flow:
1. Page loads.
2. Visitor immediately sees TarkVitark as the dominant heading.
3. Visitor sees debate-platform tagline.
4. Visitor sees coming-soon message with excited tone.
5. Visitor leaves with clear understanding: what product is + not live yet.
- Exit: Tab closed, back navigation, or idle; no in-page actions required.

Primary Journey B: Public visitor lands on splash page (mobile)
- Entry: Direct URL open in mobile browser at 375px representative viewport.
- Flow:
1. Page loads.
2. Visitor sees same three core messages in coherent order and readable layout.
3. If needed, visitor may scroll (explicitly acceptable by PO), but no interaction flow is required.
4. Visitor leaves with same understanding and sentiment goals.
- Exit: Same as desktop; no downstream flow.

System/Delivery Journey C: Static self-contained render
- Entry: Browser requests static page assets.
- Flow:
1. HTML and CSS render with no JavaScript dependency.
2. Core messaging remains available even if external resources fail (none required for function).
3. Visual treatment remains colorful/exciting using local definitions.
- Exit: Render complete; no runtime statefulness.

## 3) State Matrix

Screen: Single Coming Soon Splash Page

Desktop (1280px)
- Default/Initial Render:
1. All three core messages visible without interaction.
2. Dominant heading hierarchy intact.
3. No overflow, clipping, hidden core copy, or broken alignment.
- Loading State:
1. Browser page load only; no custom spinner/skeleton needed or expected.
2. Acceptable outcome is direct content render.
- Empty State:
1. Not applicable by design intent; if any required copy is missing, treat as failure against FR/AC.
- Error/Fallback State:
1. If optional decorative CSS treatment fails, plain readable text still visible.
2. No dependency on third-party font/assets for core meaning.
- Success State:
1. Reviewer can identify page as colorful and exciting.
2. Product identity + debate positioning + coming-soon status are clear at first glance.
- Permission/Access State:
1. Public page, no auth/role gating.
2. Permission states not applicable.

Mobile (375px)
- Default/Initial Render:
1. Core messages remain readable and visually coherent.
2. No horizontal scrolling.
3. No text overflow beyond container bounds.
- Loading State:
1. Browser-native load only; no scripted loading behavior.
- Empty State:
1. Not applicable; missing core copy is a defect.
- Error/Fallback State:
1. Same as desktop: degraded visuals acceptable, missing core messaging not acceptable.
- Success State:
1. User understands product identity and status.
2. Scrolling is acceptable per PO; above-the-fold for all copy is not required.
- Permission/Access State:
1. Public access only; no permission model.

Key Transitions
1. Request initiated -> first paint/rendered copy.
2. Rendered copy -> optional continued reading/scroll on mobile.
3. Rendered/consumed page -> session exit.
4. No JS transitions, no form-submit transitions, no route transitions.

## 4) Interaction Notes

Content and hierarchy
- Message priority must be unambiguous:
1. TarkVitark (primary heading).
2. Debate-platform tagline.
3. Coming-soon statement with exclamatory tone.
- Minor wording variation is allowed only if semantic intent is preserved.

Layout and responsive behavior
- Representative viewports are fixed for acceptance: 1280px desktop and 375px mobile.
- Desktop requires immediate visibility of key copy without interaction.
- Mobile allows scroll; must remain coherent and free of horizontal overflow.
- Single-screen/single-page structure only; no navigation constructs.

Visual direction
- Must read as energetic and colorful, not neutral/minimal.
- Minimum two distinct non-neutral colors.
- Direction is temporary and must not imply final brand system.

Technology and behavior constraints
- HTML/CSS only, zero JavaScript.
- No JS-dependent interactions, flows, or animation.
- Pure CSS motion is permitted but optional and must remain scope-safe.
- No forms, no input handling, no data/state persistence.

Accessibility and compliance
- Accessibility compliance is explicitly deferred for this slice by PO.
- Still avoid accidental unreadability that would block basic content perception in normal browsing conditions.

Dependencies
- No backend/API dependency.
- No required external fonts/assets for core function.
- Must remain functional as self-contained static deliverable.

## 8) UX Flow/State Package (for Figma handoff)

Canonical requirement summary
- Slice: coming-soon-splash-page.
- Deliverable: single static HTML/CSS splash page.
- Core UX objective: instant comprehension of product identity, product type, and pre-launch status with energetic visual sentiment.
- Platforms: desktop and mobile web at representative 1280px and 375px.

Scope boundaries carried from PRD
- In scope:
1. One page only.
2. Three required core messages in one cohesive composition.
3. Responsive coverage at 1280 and 375.
4. Colorful, excitement-oriented temporary visual direction.
5. HTML/CSS-only implementation and behavior.
- Out of scope:
1. JavaScript flows.
2. Form/input flows.
3. Navigation/multi-page flows.
4. Authenticated or role-based experiences.
5. Accessibility compliance work for this slice.
6. Final brand system definition.

Primary users, goals, and flow inventory
- Users:
1. Public visitor on desktop.
2. Public visitor on mobile.
- Goals:
1. Recognize TarkVitark immediately.
2. Understand it is a debate platform.
3. Understand it is coming soon.
4. Leave with positive excitement.
- Flow inventory:
1. Desktop arrival and instant message consumption.
2. Mobile arrival and coherent message consumption (scroll acceptable).
3. Static render/fallback behavior with no runtime dependencies.

Screen/state matrix with transitions
- Screen count: 1.
- State coverage:
1. Default render (desktop/mobile).
2. Browser-native load (no custom loading UX required).
3. Non-applicable empty/permission states documented explicitly.
4. Fallback render when optional styling degrades.
5. Success perception state tied to comprehension + excitement sentiment.
- Transitions:
1. Request -> render.
2. Render -> read/optional mobile scroll.
3. Read -> exit.
4. No scripted transitions.

2. Explicitly label output as temporary direction for this slice.

Open questions with owner status

Traceability snapshot to PRD requirements
