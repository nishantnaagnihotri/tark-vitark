### Why Native Components
M3 DS library components (`SegmentedControl`, `TextField/Outlined`, `Divider/Horizontal`) have inner frames with `layoutMode:NONE` and hardcoded dimensions (`207px`, `210px`, `320px` respectively) that the Figma Plugin API cannot override — `resize()`, `layoutSizingHorizontal`, and `primaryAxisSizingMode` all silently fail on instance internals. Native rebuilds using DS tokens are pixel-accurate and implementer-unambiguous. DS components remain correct for the approved baseline content frames.

## 2) UX Flows

### Flow A — Publish Post from Debate Screen Composer

| Step | User/System Flow |
|---|---|
| Entry | Visitor is on the existing static Debate Screen. Composer is always visible, bottom-anchored. |
| 1 | Side Selection shows M3 single-select Segmented Control (Tark default / Vitark). |
| 2 | Visitor enters Post text in Composer input. |
| 3 | Visitor selects Publish. |
| 4 | System validates Trimmed Text length 10..300 inclusive. |
| 5 | If valid, Post Publishes immediately in-page; Chronological Append at bottom. |
| Exit | Composer stays on Debate Screen for next Post. Create-Only Lifecycle in effect. |

### Flow B — Validation: whitespace-only input

| Step | User/System Flow |
|---|---|
| Entry | Input contains only whitespace. |
| 1 | Visitor selects Publish. |
| 2 | Trimmed length = 0. Publish blocked; inline error shown. |
| Exit | Visitor edits until Trimmed Text length 10..300. |

### Flow C — Validation: below minimum

| Step | User/System Flow |
|---|---|
| Entry | Trimmed Text length 1..9. |
| 1 | Visitor selects Publish. Blocked; min-length feedback shown. |
| Exit | Visitor continues editing until trimmed length ≥ 10. |

### Flow D — Validation: above maximum

| Step | User/System Flow |
|---|---|
| Entry | Trimmed Text length > 300. |
| 1 | Visitor selects Publish. Blocked; max-length feedback shown. |
| Exit | Visitor edits until trimmed length ≤ 300. |

### Flow E — Side memory and preselection behavior

| Step | User/System Flow |
|---|---|
| Entry | Debate Screen loaded; Tark selected by default. |
| 1 | Visitor changes Side Selection to Vitark or Tark. |
| 2 | System remembers last selected side for subsequent Posts until refresh. |
| Exit | Full page refresh resets Side Selection to default Tark. |

### Flow F — Mobile keyboard-open journey

| Step | User/System Flow |
|---|---|
| Entry | Mobile viewport; keyboard opened while editing Composer. |
| 1 | Composer remains visible and pinned above keyboard overlap. |
| 2 | Publish action remains reachable and operable. |
| Exit | Visitor completes Publish journey without losing Composer visibility. |

### Flow G — Refresh reset behavior

| Step | User/System Flow |
|---|---|
| Entry | One or more local Posts were published in-session. |
| 1 | Visitor performs full page refresh. |
| 2 | Debate Screen restores Baseline Static Debate Content. Session-added Posts removed. Side Selection resets to Tark. |
| Exit | New session starts from baseline state. |

## 3) State Matrix

| Surface | Default | Error | Success | Permission |
|---|---|---|---|---|
| Debate Screen with Composer | Composer visible, Tark preselected | Validation feedback (whitespace-only, <10, >300) | Valid publish appends Post at bottom | Open posting; no auth gate |
| Side Selection Segmented Control | Tark selected | No side error under enforced single-select | Last selected side retained until refresh | Available to all visitors |
| Post Text input | Empty, editable | Inline error for out-of-range Trimmed Text | Input clears after valid Publish | Available to all visitors |
| Publish action | Disabled (empty) | Disabled with feedback in context | Returns to disabled idle (input cleared) | Available to all visitors |
| Chronological Append list | Baseline Static Debate Content | Network failure OOS | New Post at bottom-most position | Publicly visible |
| Mobile keyboard-open viewport | Composer visible above keyboard overlap | Same validation states | Publish path usable without closing keyboard | No auth gate |

State transitions:
1. Default → Error when Trimmed Text is whitespace-only, below 10, or above 300.
2. Error → Default when input corrected to 10..300 inclusive.
3. Default(valid) → Success → Default (cleared, side preserved).
4. Any in-session state → Refresh Reset on full page refresh.

## 4) UI Control Contract

| Journey Step | Control Type | State Contract | Validation Feedback | Focus/Keyboard | Success Handling |
|---|---|---|---|---|---|
| Select side | M3 single-select Segmented Control (Tark / Vitark) | One side always selected. Default Tark on load. ## 13) Interaction Notes

1. Posting is inline on Debate Screen only — no FAB, no full-screen compose flow.
2. Side Selection is M3 single-select Segmented Control with exactly two options.
3. Default Tark on load; last selected side preselected after interaction until full refresh.
4. Validation uses Trimmed Text length; 10..300 inclusive only.
5. Whitespace-only is invalid; internal spaces and newlines are allowed.
6. Open posting; no auth gate. Local-only behavior in this slice.
7. Publish is immediate in-page only. Network failure handling is out of scope.
8. Chronological Append: each new Post at the very bottom.
9. Create-Only Lifecycle: no edit/delete controls in this slice.
10. Mobile keyboard-open: Composer must remain visible above overlap; Publish stays reachable.
11. Full refresh resets to Baseline Static Debate Content and removes all session-added Posts.
12. Accessibility baseline: keyboard operation, visible focus, associated error messaging, adequate contrast.
